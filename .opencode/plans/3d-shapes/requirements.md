## 1. Business Problem Statement

### Current Issue
The LiDAR system currently processes point clouds through various detection and analysis nodes (e.g., object detection, plane detection, bounding box extraction), but backend nodes have no standardized mechanism to communicate detected geometric shapes back to the frontend for visualization. Additionally, the system needs to handle shapes from multiple concurrent nodes while maintaining clear attribution and stable identity for each shape. Developers must either:

1. **Manually overlay shapes** in frontend code based on hardcoded assumptions
2. **Ignore shape data** entirely, losing valuable visualization context
3. **Implement custom per-node visualization logic**, creating maintenance burden
4. **Manually track shape identity** across frame updates, leading to visual flickering and state loss

### Business Impact
- **Developer Productivity**: No clear path to visualize detection results (bounding boxes, labels, planes)
- **System Usability**: Operators cannot see what the DAG is detecting without reading log files
- **Debugging Complexity**: Lack of visual feedback makes it difficult to validate detection algorithm accuracy
- **Feature Velocity**: Each new detection node requires custom frontend integration work
- **Multi-Node Coordination**: No mechanism to distinguish shapes from different nodes or track shape identity across frames

## 2. Feature Requirements

### 2.1 Core Functional Requirements

- [x] **R1**: Backend DAG nodes MUST be able to include shape metadata in their processing results
- [x] **R2**: NodeManager MUST automatically detect shape data in node responses and broadcast it to frontend clients
- [x] **R3**: System MUST support three shape types in the initial implementation:
  - **Cube**: Aligned bounding boxes (derived from Open3D AxisAlignedBoundingBox)
  - **Label**: Billboard text anchored to 3D coordinates (always faces camera)
  - **Plane**: Planar surface representation (format TBD by architecture)
- [x] **R4**: Shape data MUST be transmitted via a **single shared WebSocket topic** (`shapes`) using JSON format
- [x] **R5**: Each shape object MUST include:
  - `id`: Globally-unique, stable identifier generated by backend
  - `node_name`: Name of the originating DAG node for attribution
  - Shape-specific geometry data (coordinates, dimensions, etc.)
  - Optional metadata (color, label text, etc.)
- [x] **R6**: Shape data format MUST be standardized across all node types (no per-node custom schemas)
- [ ] **R7**: Frontend MUST render shapes as display-only overlays (no user interaction in initial release)
- [ ] **R8**: Frontend MUST be able to group, filter, or color shapes by originating `node_name`

### 2.2 Performance Requirements

- [ ] **R9**: System MUST maintain 60 FPS rendering with up to **50 shapes per frame** (across all nodes combined)
- [ ] **R10**: Shape rendering overhead MUST NOT exceed 5% of total frame time
- [ ] **R11**: Shared shape WebSocket topic MUST NOT introduce blocking operations on the DAG orchestrator
- [ ] **R12**: Shape updates MUST be synchronized with point cloud frames (updated on each node response, but not cleared globally unless all nodes finish)

### 2.3 Data Format & Protocol Requirements

- [ ] **R13**: Shape data MUST use a single shared WebSocket topic: `shapes` (not per-node topics)
- [ ] **R14**: Shape messages MUST follow JSON format (human-readable for debugging)
- [ ] **R15**: Each shape object MUST include a stable, globally-unique `id` field (e.g., UUID or `{node_name}_{shape_index}_{timestamp}`)
- [ ] **R16**: Each shape object MUST include a `node_name` field identifying the originating DAG node
- [ ] **R17**: Backend MUST ensure shape IDs remain stable across frame updates for the same logical shape (enables frontend to track changes/animations)
- [ ] **R18**: All shape coordinates MUST use the same coordinate system as the point cloud data (world/sensor frame consistency)
- [ ] **R19**: Shape lifecycle MUST support incremental updates (nodes can add/update their shapes without clearing shapes from other nodes)

### 2.4 Error Handling Requirements

- [ ] **R20**: Invalid shape data MUST be silently skipped with console logging (no frontend crashes)
- [ ] **R21**: Shape rendering failures MUST NOT block point cloud rendering
- [ ] **R22**: Shared shape WebSocket topic disconnections MUST be handled gracefully (fallback to point cloud-only mode)
- [ ] **R23**: Duplicate shape IDs (same `id` from multiple nodes) MUST be detected and logged as warnings (last-write-wins or rejected)

## 3. User Stories

### 3.1 Primary User Stories

**As a** DAG node developer (backend)  
**I want** a simple API to attach uniquely-identified bounding boxes to my detection results  
**So that** frontend automatically visualizes what my algorithm detected, and can track each shape's state across frames without custom integration code

**As a** system operator  
**I want** detected objects to display with labeled bounding boxes in the 3D viewer  
**So that** I can visually verify detection accuracy without reading log files or CSV exports

**As a** frontend developer  
**I want** a standardized shape data schema that works across all node types and includes stable IDs plus node attribution  
**So that** I can build one shape rendering pipeline with persistent state tracking instead of per-node custom logic

**As a** calibration engineer  
**I want** plane detection nodes to render the detected plane surface in 3D space  
**So that** I can verify alignment quality visually during calibration workflows

### 3.2 Edge Case Stories

**As a** detection node processing noisy data  
**I want** invalid shape data to be skipped gracefully  
**So that** occasional bad frames don't crash the entire visualization pipeline

**As a** frontend rendering engine under load  
**I want** shape rendering to degrade gracefully if too many shapes are sent  
**So that** the system remains responsive even if backend sends unexpected volume

**As a** developer debugging WebSocket issues  
**I want** shape data transmitted in human-readable JSON format with clear node attribution  
**So that** I can inspect WebSocket traffic with browser DevTools and identify which node sent which shapes during troubleshooting

**As a** system with multiple concurrent detection nodes  
**I want** each shape to include its originating node name  
**So that** frontend can color-code or filter shapes by node for visual debugging and analysis

## 4. Acceptance Criteria

### 4.1 Backend Integration Acceptance Criteria

- [x] **AC1**: NodeManager provides a helper method for nodes to attach shape data to their output (e.g., `attach_shapes(node_id, shapes: List[ShapeData])`)
- [x] **AC2**: When a node includes shape data, NodeManager broadcasts it to the shared WebSocket topic `shapes`
- [x] **AC3**: Backend automatically generates stable, globally-unique IDs for each shape (e.g., `{node_name}_{shape_type}_{index}` or UUID)
- [x] **AC4**: Backend automatically injects `node_name` field into each shape object before broadcasting
- [x] **AC5**: Backend validates shape schema structure before broadcasting (reject malformed data with error log)
- [ ] **AC6**: Shared `shapes` topic lifecycle: created on first shape emission, persists across node additions/removals, cleaned up only when no nodes are producing shapes

### 4.2 Shape Type Acceptance Criteria

#### Cube (Bounding Box)
- [ ] **AC7**: Cube shapes include minimum and maximum corner coordinates (axis-aligned box)
- [ ] **AC8**: Cube shapes include `id` (backend-generated) and `node_name` (originating node) fields
- [ ] **AC9**: Cube shapes support optional color/label metadata
- [ ] **AC10**: Frontend renders cubes as wireframe boxes by default

#### Label (Billboard Text)
- [ ] **AC11**: Label shapes include 3D anchor position and text string
- [ ] **AC12**: Label shapes include `id` (backend-generated) and `node_name` (originating node) fields
- [ ] **AC13**: Frontend renders labels as screen-space text that always faces the camera (billboard behavior)
- [ ] **AC14**: Labels use basic fixed font styling (size, color, background TBD by architecture)

#### Plane (Surface)
- [ ] **AC15**: Plane shape definition is determined by architecture (point+normal or 4-corner or mesh)
- [ ] **AC16**: Plane shapes include `id` (backend-generated) and `node_name` (originating node) fields
- [ ] **AC17**: Frontend renders planes as semi-transparent surfaces (visible from both sides)

### 4.3 Frontend Rendering Acceptance Criteria

- [ ] **AC18**: Frontend subscribes to the shared `shapes` WebSocket topic when DAG is active
- [ ] **AC19**: Frontend uses shape `id` field to maintain persistent state (track shape changes across frames, avoid re-creating geometry unnecessarily)
- [ ] **AC20**: Frontend uses `node_name` field to group or color shapes by originating node
- [ ] **AC21**: Shape rendering updates incrementally (new shapes added, removed shapes deleted, updated shapes modified in-place based on `id` matching)
- [ ] **AC22**: Rendering maintains 60 FPS with 50 shapes (from multiple nodes) simultaneously visible
- [ ] **AC23**: Invalid shape data logs warning to console and skips rendering that specific shape (other shapes render normally)
- [ ] **AC24**: Duplicate shape IDs detected and handled (either last-write-wins or logged warning)

### 4.4 WebSocket Protocol Acceptance Criteria

- [ ] **AC25**: Shape topic uses fixed name: `shapes` (shared by all nodes)
- [ ] **AC26**: Shape messages use JSON format (not binary LIDR protocol)
- [ ] **AC27**: Each shape message includes array of shape objects, each with required `id` and `node_name` fields
- [ ] **AC28**: WebSocket close code 1001 ("Going Away") is sent when shape topic is removed (consistent with existing topic cleanup protocol)
- [ ] **AC29**: Frontend handles shape topic disconnection without crashing point cloud rendering

### 4.5 Shape Identity & Deduplication Acceptance Criteria

- [ ] **AC30**: Backend generates stable shape IDs that persist across frames for the same logical shape (e.g., tracked object maintains same ID until it disappears)
- [ ] **AC31**: Shape IDs are globally unique (no collisions between nodes or shape types)
- [ ] **AC32**: Frontend detects duplicate shape IDs and handles gracefully (logs warning, uses last-write-wins or first-write-wins strategy)
- [ ] **AC33**: Frontend can query all shapes from a specific node using `node_name` filter
- [ ] **AC34**: When a node is removed from DAG, its shapes are automatically removed from the shared topic (backend sends deletion message or frontend times out stale shapes)
### 4.6 Integration Acceptance Criteria

- [ ] **AC35**: Existing point cloud rendering remains unaffected when shape feature is inactive (no performance regression)
- [ ] **AC36**: Performance monitoring dashboard shows shape rendering overhead separately from point cloud rendering
- [ ] **AC37**: All existing DAG nodes continue working without modification (backward compatible)
- [ ] **AC38**: At least one example node (e.g., mock object detection) demonstrates full shape integration end-to-end with stable IDs and node attribution

## 5. Out of Scope

### 5.1 Explicitly Not Included

- **Shape Interactivity**: No clicking, selecting, dragging, or editing shapes in this release (display-only)
- **Shape Persistence Across Sessions**: Shapes do not persist between application restarts (memory-only state)
- **Shape History/Replay**: No timeline scrubbing or historical shape visualization
- **Custom Shape Types**: Only Cube, Label, and Plane are supported (no arrows, cylinders, meshes, etc.)
- **Rich Text Rendering**: Labels are plain text only (no HTML, markdown, or custom fonts)
- **Shape Styling API**: Limited style customization (color/size controlled by architecture defaults)
- **Shape Filtering UI**: No frontend controls to show/hide specific shape types or categories (node-based filtering acceptable)
- **Performance Optimization for High Volume**: System targets 50 shapes/frame across all nodes (not 200+)
- **LIDR Protocol Extension**: Shapes use separate JSON WebSocket topic (not embedded in binary protocol)
- **Cross-Node Shape Correlation**: No mechanism to automatically link or merge shapes from multiple nodes (e.g., fused detections)
- **Per-Node Shape Topics**: Shapes from all nodes broadcast over single shared `shapes` topic (not `{node_topic}_shapes` pattern)

### 5.2 Future Considerations

- **Interactive Editing**: Click to select, delete, or modify shapes in 3D viewer
- **Shape Categories/Layers**: Group shapes by type with toggle visibility controls
- **Advanced Shapes**: Arrows (velocity vectors), cylinders (poles), custom meshes
- **Styled Labels**: Font family, size, color, background, HTML support
- **Shape Animation**: Smooth transitions when shapes update between frames
- **High-Volume Optimization**: Instanced rendering for 200+ cubes/frame

## 6. Dependencies & Assumptions

### 6.1 Technical Dependencies

- **Backend**: NodeManager must support shape data attachment with automatic ID generation and node attribution
- **Backend**: NodeManager must broadcast shapes to a single shared `shapes` WebSocket topic
- **Frontend**: Three.js rendering pipeline must support wireframe boxes, billboard sprites, and plane geometry
- **Frontend**: Shape rendering must track shape identity via `id` field for persistent state management
- **WebSocket Infrastructure**: Shared topic lifecycle management (topic persists across node changes, cleaned up when no nodes produce shapes)
- **Performance Monitoring**: Existing metrics system should track shape rendering time separately

### 6.2 Business Assumptions

- 50 shapes per frame (across all nodes) is sufficient for initial object detection use cases (10-20 objects typical per node)
- Display-only shapes provide sufficient value (interaction can be added later)
- Plain text labels are acceptable (styled text is not a blocker for MVP)
- Incremental shape updates (via stable IDs) are preferred over full frame clearing
- Single shared topic architecture is simpler and more scalable than per-node topics
- Node attribution via `node_name` field provides sufficient context for debugging and visualization

### 6.3 Architectural Constraints

- Shape coordinate system MUST match point cloud coordinate system (no transform mismatch)
- Shape WebSocket topic uses single shared `shapes` topic (not per-node topics)
- Each shape MUST include backend-generated `id` and `node_name` fields
- Shape schema is defined by architecture team (backend/frontend must agree on JSON structure)
- Shape IDs MUST be stable across frame updates for the same logical shape (enables frontend state persistence)

## 8. Rollout Plan

### 8.1 Implementation Phases

1. **Phase 1 (Architecture)**: Define JSON shape schema with `id`, `node_name`, and coordinate system conventions (`api-spec.md`)
2. **Phase 2 (Backend)**: Implement NodeManager shape attachment API with automatic ID generation and node attribution, plus shared `shapes` WebSocket topic broadcasting
3. **Phase 3 (Frontend)**: Build Three.js shape rendering pipeline (Cube, Label, Plane) with `id`-based state management and `node_name` filtering/grouping
4. **Phase 4 (Integration)**: Create example detection node demonstrating end-to-end shape workflow with stable IDs
5. **Phase 5 (QA)**: Performance testing, error handling validation, deduplication testing, visual regression testing

### 8.2 Testing Strategy

- **Unit Testing**: 
  - Backend: Shape schema validation, WebSocket topic creation/cleanup, ID generation uniqueness, node attribution injection
  - Frontend: Shape geometry generation, billboard text rendering, `id`-based state tracking, `node_name` filtering
- **Integration Testing**: 
  - End-to-end: Node sends shapes → WebSocket → Frontend renders with correct attribution
  - Error cases: Malformed JSON, invalid coordinates, missing `id`/`node_name` fields, duplicate IDs
  - Multi-node: Multiple nodes sending shapes concurrently to shared topic
- **Performance Testing**: 
  - Render 50 shapes (from 3+ nodes) at 60 FPS for 60 seconds
  - Memory leak testing (shape objects cleaned up on updates, no accumulation)
  - State management efficiency (shape updates via `id` matching don't re-create geometry)
- **Visual Regression Testing**: 
  - Cube wireframes render at correct positions/sizes with node-specific colors
  - Labels face camera from all view angles with correct node attribution
  - Planes render as semi-transparent surfaces
  - Multiple nodes' shapes visible simultaneously without interference

## 11. Open Questions for Architecture Team

### To Be Resolved in `technical.md`

1. **Plane Shape Definition**: 
   - Infinite plane (point + normal vector) with bounded rendering region?
   - Finite quad (4 corner coordinates)?
   - Triangulated mesh?

2. **JSON Schema Structure**:
   - Top-level message format (array of shapes vs. categorized by type)?
   - Shape ID generation strategy (UUID vs. composite key like `{node_name}_{type}_{index}`)?
   - Coordinate precision (float32 vs. float64)?
   - Optional vs. required fields for color, label, metadata?

3. **WebSocket Topic Lifecycle**:
   - When is shared `shapes` topic created (on first shape emission or on DAG start)?
   - When is it cleaned up (when last node stops producing shapes or on DAG shutdown)?
   - How are node-specific shape removals handled (explicit deletion messages or timeout-based)?

4. **Label Rendering Details**:
   - Fixed font size in pixels or world-space scaling?
   - Default text color (white, black, or derived from background)?
   - Background box rendering (transparent, opaque, or none)?

5. **Error Handling Scope**:
   - Should backend reject ALL shapes in a message if ONE is invalid, or skip only the bad shape?
   - Should frontend display error count badge on DAG node when shapes are skipped?
   - How to handle duplicate shape IDs (last-write-wins, first-write-wins, or reject with error)?

6. **Performance Monitoring Integration**:
   - Should shape rendering metrics appear in existing dashboard or new dedicated panel?
   - Which metrics to track: render time, shape count, memory usage, state update efficiency?

7. **Multi-Node Shape Management**:
   - Should frontend provide UI controls to filter shapes by `node_name`?
   - Should shapes from different nodes use different default colors automatically?
   - How to handle stale shapes when a node crashes (timeout vs. explicit cleanup)?
